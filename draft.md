# 文档收尾专用草稿

## 背景

当前仓库的产品实现已经基本完成，核心验证结果如下：

- `npm run demo` 可稳定产出完整的 deterministic demo report。
- `npm run validate:data` 通过，数据规模满足当前实现要求。
- `npm test` 通过，当前为 `20/20`。
- `npm run benchmark` 通过。

当前剩余问题主要不是产品代码，而是 RLCR 文档闭环没有收干净。最近几轮 review 反复卡在 `docs/agent-usage.md` 对“当前轮 artifact”的记录总比实际 review 结果慢一步，导致文档对齐问题在 Round 7/8/9 重复出现，并最终触发 `STOP`。

## 本轮新目标

新会话应只处理文档，不做产品功能开发，不改业务逻辑，不扩展测试范围，不重构代码。

目标是把仓库中的交付文档、RLCR 文档说明和最终结论整理成一个一致、可交接、可解释的状态，使后续 review 不再因为“当前轮 artifact 自引用滞后”而陷入循环。

## 范围

### 允许修改

- `docs/agent-usage.md`
- `README.md` 中与交付说明、验证状态、文档索引相关的少量内容
- 新增或更新必要的交接文档、说明文档
- 如确有必要，可新增一个更稳定的文档记录策略说明文件

### 不允许修改

- `src/`
- `public/`
- `tests/`
- `scripts/`
- 任何产品实现代码
- 历史 RLCR 状态文件，例如 `.humanize/.../state.md`
- 历史 goal tracker 内容，除非新的 loop 明确要求且流程允许

## 核心问题定义

需要解决的不是“功能没完成”，而是“文档记录策略不稳定”：

1. `docs/agent-usage.md` 目前曾因记录当前轮状态而出现时序问题。
2. review 文件一旦落盘，文档里“review result pending”之类的表述就会过时。
3. 如果继续记录“当前轮进行中状态”，很容易再次重复同样的问题。

## 建议策略

建议新会话采用下面的稳定策略之一，并在文档中明确写死：

- 只记录“已完成轮次”，不记录当前正在进行的轮次。
- 或者把“当前轮”单独标成 transient/in-progress 区块，并明确说明它不作为验收依据。
- 或者增加“post-review docs refresh”作为固定步骤，但不要再提前宣称当前轮已经闭合。

优先推荐第一种：只记录已完成轮次。

## 期望交付

新会话应至少完成以下产物：

1. 修正 `docs/agent-usage.md`，让其对 RLCR 轮次和 artifact 的叙述不再依赖当前轮自引用。
2. 如果需要，在 `README.md` 中补一段简短说明，澄清：
   - 产品功能已验证通过；
   - 当前残留问题是文档流程问题，不是业务逻辑故障；
   - 当前受限环境下无法实际监听 `127.0.0.1` 端口。
3. 保留一个简明交接说明，方便后续 reviewer 理解为什么新的文档策略更稳定。

## 验收标准

- AC-1: 文档修复后，`docs/agent-usage.md` 不再因为“当前轮 review 文件刚生成”而立即过时。
- AC-2: 文档叙述能清楚区分“产品实现状态”和“RLCR 文档流程状态”。
- AC-3: 文档中不再错误暗示产品代码仍存在未验证的核心缺陷。
- AC-4: 新会话的目标范围被明确限制为文档修复，不包含产品代码改动。
- AC-5: 交接说明足够简洁，新会话无需重新依赖旧 `plan.md` 的完整上下文也能继续。

- 可以基于本 `draft.md` 重新生成一个更短的 docs-only plan。
- 新 loop 应优先面向文档收尾，而不是重新审视整个产品实现。
