# 个性化旅游系统设计与实现计划

## 目标说明

将草稿中的课程设计要求整理为一个可实施、可验收、可演示的个性化旅游系统方案。系统应覆盖旅游前推荐、旅游中路线规划与场所查询、旅游后日记管理与交流、美食推荐等完整流程，并满足课程对数据规模、核心数据结构与算法、自主实现、文档交付、Git 版本控制、回归测试和测试优先开发的要求。

基于当前仓库只有规划材料这一现状，本计划默认需要从零建立项目结构、测试体系、数据集、算法模块、演示界面和课程文档。草稿中出现的数量约束例如“至少”“不少于”均按硬性验收要求处理，而“建议爬取真实地图数据”“建议尝试使用智能体”等按优先建议处理。

## 假设与澄清

- 草稿中的“至少”“不少于”均视为硬性验收门槛，不作为趋势性指标处理。
- “景区和校园内部可以一致”解释为可以复用同一套图模型、建筑模型和设施模型，但演示数据中仍应保留景区与校园两类场景标签。
- “新闻、Q&A、视觉材料”这一表述语义不完整，本计划将其归入需求分析与演示支撑材料，而不额外扩展为独立核心功能。
- “AIGC 旅游动画生成”按可演示、可复验的动态内容生成能力处理，允许实现为短视频、动态分镜或其它可播放产物。

## 验收标准

以下验收标准采用 TDD 思路组织，每项都包含可确定验证的正向与反向测试。

- AC-1: 系统数据模型与基础数据集满足课程最低规模要求，并能被程序稳定加载与校验。
  - 正向测试（应通过）：
    - 导入后的景区/校园实体总数不少于 200 个，校验脚本返回成功。
    - 每个被演示的景区或校园都具备道路图、建筑物、服务设施、边信息和基础元数据。
    - 建筑物数量不少于 20 个，服务设施种类不少于 10 种，服务设施实例不少于 50 个，道路边数不少于 200 条。
    - 用户样本数不少于 10 人，系统可以完成基础登录或身份切换演示。
  - 反向测试（应失败）：
    - 当景区/校园数量少于 200 个时，数据校验命令必须报错并指出缺失项。
    - 当图中边数不足或设施分类不完整时，系统不得进入可演示状态。
    - 当输入数据缺少节点坐标、建筑类别或服务设施类型时，导入流程必须拒绝不完整记录。

- AC-2: 旅游推荐模块支持按热度、评分和个人兴趣进行目的地推荐，并支持基于名称、类别、关键字的查询与结果排序。
  - 正向测试（应通过）：
    - 给定带有兴趣标签的用户画像，系统能够返回按规则排序的景点/学校推荐结果。
    - 当用户只查看前 10 个结果时，系统使用非完全排序策略仍能稳定返回正确 Top-K。
    - 按名称、类别或关键字检索时，若命中多项，结果可按热度和评分重新排序。
    - 动态修改热度或评分后，推荐列表能反映最新数据而不需要全量重建。
  - 反向测试（应失败）：
    - 查询条件为空且未指定默认策略时，接口不得返回无约束的海量结果。
    - 当排序字段非法或权重配置越界时，请求必须被拒绝并返回明确错误。
    - 不允许直接依赖数据库现成排序替代核心排序算法实现。
  - AC-2.1: 推荐与查询相关核心算法必须基于自定义数据结构实现。
    - 正向测试（应通过）：
      - 排序模块可独立运行并输出复杂度统计或基准结果。
      - 查找索引支持精确匹配、关键字匹配或前缀匹配中的至少一种高效实现。
    - 反向测试（应失败）：
      - 若关闭自定义算法模块后系统仍能完成全部推荐与查询，则说明实现不符合课程要求。

- AC-3: 旅游路线规划模块支持单目标最短路径、多目标闭环路径、地图展示和路径可视化。
  - 正向测试（应通过）：
    - 输入当前位置与单个目标点后，系统能返回一条有效路径、总距离和经过节点序列。
    - 输入多个目标点后，系统能生成从当前位置出发、访问所有目标并返回起点的可行最优或约束内最优路线。
    - 地图界面能够高亮路径、起终点和关键转折点。
    - 不存在路径时，系统能返回不可达状态而不是错误路径。
  - 反向测试（应失败）：
    - 当目标点不存在于图中时，系统必须拒绝规划请求。
    - 当图数据断裂时，系统不得伪造联通路径。
    - 当多目标请求包含重复或非法节点时，系统必须进行去重或报错，不得静默忽略。
  - AC-3.1: 路线规划必须支持最短距离、最短时间和多交通工具最短时间三种策略。
    - 正向测试（应通过）：
      - 最短距离策略在统一权重图上返回距离最小的路线。
      - 最短时间策略可结合拥挤度和理想速度计算总代价。
      - 校园场景下支持步行与自行车，景区场景下支持步行与电瓶车，并遵守道路类型限制。
      - 多交通工具策略允许混合换乘时，返回的路线和时间计算结果一致。
    - 反向测试（应失败）：
      - 当自行车被规划到非自行车道路，或电瓶车被规划到非电瓶车路线时，请求必须失败或自动排除非法边。
      - 当拥挤度小于等于 0 或大于 1 时，数据校验必须报错。
  - AC-3.2: 系统支持室内导航。
    - 正向测试（应通过）：
      - 对至少一个教学楼或馆内建筑，系统可完成门口到电梯、电梯跨楼层、楼层到房间的路径规划。
      - 室内外图切换时，界面和路径结果能正确提示场景边界。
    - 反向测试（应失败）：
      - 当楼层连接关系缺失时，系统不得生成跨楼层路径。

- AC-4: 场所查询模块支持在景区/校园内部按网络距离查找附近服务设施，并支持类别过滤与按距离排序。
  - 正向测试（应通过）：
    - 选定景点或场所后，系统可返回一定范围内的超市、卫生间等设施，并按图上真实可达距离排序。
    - 用户可按设施类别过滤结果，且结果数量和排序与过滤条件一致。
    - 用户输入类别名称查询附近设施时，系统能返回符合条件的排序结果。
  - 反向测试（应失败）：
    - 不能使用直线距离代替道路网络距离。
    - 当类别不存在时，系统不得返回无关设施。
    - 当半径参数为负数或目标位置缺失时，请求必须报错。

- AC-5: 旅游日记管理模块支持多媒体日记创建、统一存储、浏览、评分与推荐。
  - 正向测试（应通过）：
    - 用户可创建包含文字、图片和视频元数据的旅游日记，并可被持久化保存。
    - 系统能展示所有用户日记，浏览量可累计，评分可更新平均值或聚合值。
    - 日记推荐支持按热度、评分和个人兴趣排序，并可使用 Top-K 策略优化前 10 条结果。
  - 反向测试（应失败）：
    - 当日记缺少作者、目的地或正文等必填字段时，保存操作必须失败。
    - 同一用户重复提交非法评分或越界评分时，系统必须拒绝。
    - 删除或下线日记后，不得继续出现在推荐列表中。

- AC-6: 旅游日记交流模块支持按目的地排序查询、按名称精确检索、全文检索、无损压缩存储和基于图片/文字的旅游动画生成。
  - 正向测试（应通过）：
    - 输入旅游目的地后，系统可返回相关日记，并按热度和评分排序。
    - 输入日记名称时，系统可进行高效精确查询。
    - 输入正文关键字时，全文检索模块可返回相关日记及命中信息。
    - 日记正文或媒体索引可执行无损压缩与解压，解压后内容与原文一致。
    - 输入景点/学校照片与文字描述后，系统可生成可展示的旅游动画、短视频或分镜式动态内容。
  - 反向测试（应失败）：
    - 当精确名称不存在时，系统不得返回近似但不相同的标题作为精确查询结果。
    - 压缩后若无法恢复原始内容，测试必须失败。
    - 当 AIGC 输入缺少必要素材时，生成流程必须返回可解释错误而不是空文件。
  - AC-6.1: 与文本搜索、压缩、AIGC 相关的能力必须可独立验证。
    - 正向测试（应通过）：
      - 文本搜索索引构建、检索、删除和更新均有自动化测试。
      - 压缩算法对典型日记样本具备可重复的压缩率统计。
      - AIGC 模块通过接口抽象隔离模型供应方，测试环境可使用 mock 生成器验证流程。
    - 反向测试（应失败）：
      - 若关闭搜索索引后仍依赖数据库全文索引完成核心功能，则不符合课程要求。
      - 若 AIGC 模块仅返回占位文本且不生成任何可演示产物，则验收失败。

- AC-7: 美食推荐模块支持按热度、评分和距离排序，按菜系过滤，以及基于内容的模糊查询。
  - 正向测试（应通过）：
    - 在选定景点或学校后，系统可按热度、评分和距离组合排序推荐美食。
    - 用户可按菜系过滤结果，排序结果仍然稳定。
    - 输入美食名称、菜系、饭店名或窗口名后，系统可完成模糊查询并对多项结果排序。
    - 前 10 个美食结果可通过非完全排序算法返回。
  - 反向测试（应失败）：
    - 当模糊查询阈值配置错误导致所有结果都命中时，系统必须报错或回退到安全阈值。
    - 当距离字段缺失时，不得继续使用需要距离排序的策略。

- AC-8: 项目交付物完整，包含课程要求文档、测试结果、示例运行结果和用户说明，并能支撑最终演示。
  - 正向测试（应通过）：
    - 需求分析或演示材料中明确说明至少 1 项创新点，并能解释其解决了现有旅游系统未覆盖或覆盖不足的实际需求。
    - 仓库中包含任务描述、需求分析、总体方案、数据结构说明与数据字典、模块设计、应用范例与测试说明、评价与改进意见、用户使用说明。
    - 若课程教师要求配套展示材料，仓库中包含新闻稿、Q&A 或视觉材料中的对应产物。
    - 提供可重复执行的回归测试命令，且在提交前可全部通过。
    - 提供至少一组完整演示脚本，覆盖推荐、路线规划、场所查询、日记、日记交流和美食推荐六类功能。
    - 若使用智能体辅助开发，需记录智能体用途、环境和链接。
  - 反向测试（应失败）：
    - 若只实现通用旅游信息展示而没有明确创新点说明，不得判定为达到课程要求。
    - 缺少任一课程要求文档时，不得判定为可交付版本。
    - 演示脚本若无法从空数据库或初始化数据集稳定复现，则验收失败。

## 范围边界

### 上界（最大可接受范围）

在满足所有硬性验收要求的前提下，系统可以实现为一个完整的前后端应用，包含：

- 真实或接近真实的景区/校园图数据采集与清洗脚本
- 面向用户的 Web 界面或桌面界面
- 可视化地图与室内外导航联动
- 推荐、检索、路径规划、压缩、全文搜索、AIGC 生成等核心算法的性能对比与实验报告
- 规范化课程文档、演示脚本和回归测试体系

### 下界（最小可接受范围）

最低可交付实现也必须同时满足以下条件，不能只做其中一部分：

- 完成草稿列出的六大功能域：旅游推荐、路线规划、场所查询、旅游日记管理、旅游日记交流、美食推荐
- 满足所有数量型硬性指标：200 个景区/校园、20 个建筑物、10 类服务设施、50 个设施实例、200 条边、10 个用户
- 每个功能域均有可演示界面或等价的交互入口
- 每个核心算法均为自主实现，且有自动化测试或可复验脚本
- 完成课程要求的文档交付

### 允许的实现选择

- 可以使用：TypeScript 全栈、前后端分离或单体架构、SQLite/PostgreSQL/JSON 数据存储、外部地图数据作为原始素材、外部模型或 API 辅助 AIGC 生成
- 可以使用：堆、平衡树、跳表、Trie、倒排索引、图邻接表、优先队列、Dijkstra、A*、多目标路径启发式算法、Huffman/LZ 系列无损压缩
- 不能使用：把数据库 `ORDER BY`、数据库全文索引、第三方地图路径规划结果或现成推荐引擎直接充当课程要求中的核心算法实现
- 不能使用：只做静态页面、只做命令行演示、只交文档不交系统、只做单一场景而缺失其它功能域

## 可行性提示与建议

> 本节是实现建议，不是额外强制要求。若后续实现过程中发现更合适的方案，可以在不违反验收标准的前提下调整。

### 建议的系统拆分

建议先按“数据层、算法层、应用层、表现层、文档层”五部分建立仓库结构：

- `data/`
  - 原始地图、清洗脚本、种子数据、用户样本数据
- `src/domain/`
  - 景区、校园、建筑、设施、用户、日记、美食等领域模型
- `src/algorithms/`
  - 排序、查找、模糊匹配、图最短路径、多目标路径、全文检索、压缩
- `src/services/`
  - 推荐服务、路线服务、查询服务、日记服务、AIGC 编排服务
- `src/ui/`
  - 推荐页、地图页、场所查询页、日记页、美食页
- `docs/`
  - 课程要求文档、测试报告、演示说明、用户手册

### 建议的数据结构与算法映射

- 旅游推荐与美食推荐：
  - 使用自定义 Top-K 结构，例如最小堆或快速选择，避免全量排序
  - 使用倒排标签索引或 Trie 支撑兴趣、关键字和类别查询
- 路线规划：
  - 使用邻接表表示道路图
  - 使用 Dijkstra 或 A* 求单目标最短路
  - 使用状态压缩 DP、分支限界或启发式算法处理多目标闭环路径
  - 使用分层图或带场景标签的图处理室内外切换
- 场所查询：
  - 基于图距离而非几何直线距离，必要时复用最短路结果缓存
- 日记交流：
  - 使用倒排索引处理全文检索
  - 使用 Trie、哈希索引或平衡树处理标题精确查询
  - 使用 Huffman 或 LZ 方案处理无损压缩
  - AIGC 模块通过统一接口接入模型，便于 mock 测试与后续替换

### 关键实现取舍

- AIGC 动画生成是最不稳定的部分，建议将“生成一段可展示的动态内容”作为底线，不把画质质量写进硬指标
- 室内导航建议先选 1 个建筑做完整样板，再扩展到更多建筑
- 若真实地图数据清洗成本过高，可先用真实数据加人工补全的方式保证图质量与数量要求
- 若时间紧张，优先保证核心算法、数据规模、测试和演示链路完整，再增强视觉效果

## 依赖关系与实施顺序

### 里程碑

1. 里程碑一：需求固化与仓库初始化
   - 阶段 A：将本计划拆成 issue、目录结构和模块边界
   - 阶段 B：初始化测试框架、格式化工具、提交前回归脚本和基础 CI
   - 阶段 C：明确课程文档目录与命名规范

2. 里程碑二：数据建模与种子数据落地
   - 阶段 A：定义景区/校园图、建筑、设施、用户、日记、美食的数据模型和数据字典
   - 阶段 B：构建或导入满足最小规模的数据集
   - 阶段 C：实现数据校验脚本，确保硬性数量约束自动检查

3. 里程碑三：推荐、检索与排序能力
   - 阶段 A：完成旅游推荐与目的地查询算法
   - 阶段 B：完成美食推荐与模糊检索算法
   - 阶段 C：补齐性能对比与 Top-K 正确性测试

4. 里程碑四：图算法与导航可视化
   - 阶段 A：完成单目标、多目标、时间策略、交通工具约束的路径规划
   - 阶段 B：完成场所查询与附近设施排序
   - 阶段 C：完成地图展示、路径高亮和室内导航样板

5. 里程碑五：日记系统、搜索、压缩与 AIGC
   - 阶段 A：完成日记 CRUD、浏览量、评分和推荐
   - 阶段 B：完成目的地查询、标题精确查找、全文检索
   - 阶段 C：完成无损压缩和 AIGC 动画生成流程

6. 里程碑六：文档、回归和最终演示
   - 阶段 A：整理课程要求文档与算法说明
   - 阶段 B：补齐回归测试、示例结果、用户手册和改进建议
   - 阶段 C：形成最终演示脚本并做完整彩排

### 关键依赖

- 没有稳定数据模型，就无法稳定实现推荐、查询、图算法和文档中的数据字典
- 没有数据校验脚本，就无法保证课程中的数量型硬约束始终满足
- 路线规划与场所查询共享道路图和设施索引，应优先统一图表示
- 日记推荐、美食推荐和旅游推荐可共享排序与索引基础设施
- AIGC 生成依赖日记媒体输入与素材管理，应放在日记系统基本可用之后

## 实施说明

### 开发流程要求

- 使用 Git 进行版本控制，小步提交
- 每次提交前运行回归测试
- 按测试优先原则开发核心算法与关键业务流程
- 对新增算法模块同时提交正确性测试和性能观察结果

### 文档交付清单

- 软件开发任务描述
- 功能需求说明与分析
- 新闻稿、Q&A、视觉材料等展示支撑材料（若教师要求）
- 总体方案设计说明
- 智能体开发环境与智能体说明
- 数据结构说明与数据字典
- 各模块设计说明
- 应用范例执行结果及测试情况说明
- 评价和改进意见
- 用户使用说明

### 代码约束

- 代码与注释中不要出现“AC-”“里程碑”“阶段”等计划术语
- 领域对象、算法模块和界面组件应使用业务语义命名
- 核心算法必须保留可单独运行或可单独测试的入口

### 语言统一说明

本计划统一使用中文撰写，验收编号保留 `AC-X` 形式以方便引用。
